# Project overview
- 이 프로젝트는 대출 계산기 웹사이트다.
- 목표는 사용자가 쉽고 빠르게 대출 상환 결과를 확인하게 하는 것이다.
- 모바일 사용자 경험을 우선한다.

# Working style
- 사용자는 방향과 우선순위만 결정한다.
- 구현은 Claude가 주도한다.
- 큰 변경 전에는 짧은 계획을 먼저 제시한다.
- 작업 후에는 수정한 파일, 변경 이유, 실행 방법을 설명한다.

# UI and product rules
- 디자인은 깔끔하고 신뢰감 있게 유지한다.
- 과한 애니메이션과 불필요한 장식은 피한다.
- 결과 화면은 가독성과 전환성을 우선한다.
- CTA 문구는 과장형보다 명확한 정보형을 선호한다.
- 모바일 화면에서 입력, 결과, 버튼 흐름이 자연스럽게 이어져야 한다.

# Code rules
- 기존 기능을 깨지 않는 방향으로 수정한다.
- 가능하면 기존 파일 구조를 존중하고, 필요할 때만 구조를 크게 바꾼다.
- 재사용 가능한 컴포넌트를 우선 사용한다.
- 변경 전 영향 범위를 짧게 설명한다.

# Build and validation
- 변경 후에는 먼저 로컬에서 build 가능 여부를 확인한다.
- build 명령은 `npm run build`를 사용한다.
- build가 실패하면 원인을 수정한 뒤 다시 확인한다.

# Git workflow
- 내가 "배포해줘"라고 하면 먼저 `npm run build`를 실행한다.
- build가 성공하면 `git add .`, 적절한 커밋 메시지로 commit, 그리고 `git push`까지 진행한다.
- build가 실패하면 배포하지 말고 원인을 수정한다.
- push 후에는 GitHub와 연결된 Vercel에서 자동 배포가 시작된다고 설명한다.

# Communication
- 선택지가 있으면 2~3개 안을 먼저 제시하고 가장 현실적인 안을 추천한다.
- 설명은 코드보다 결과 중심으로 이해하기 쉽게 한다.
- 내가 확인해야 할 포인트가 있으면 짧게 알려준다. 이거는 어디 붙여넣어?
